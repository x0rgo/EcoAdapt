# EcoAdapt Auto-Flash Integration Guide

This explains how to wire the new firmware + auto-flash system into your
existing Flask app.

## Files added

```
firmware/
├── pod/pod.ino                       # Pod firmware (XIAO C3 #1)
├── bridge/bridge.ino                 # Bridge firmware (XIAO C3 #2) — 3-tier provisioning
├── web_flasher/
│   ├── flash.html                    # Web Serial flasher (Tier 1: NVS-staged WiFi)
│   ├── pair.html                     # Web Bluetooth pairing (Tier 2: BLE)
│   └── flash_routes.py               # Flask blueprint with /flash, /pair, /api/firmware/*
└── build_scripts/
    ├── setup.sh                      # One-time arduino-cli setup
    └── build.sh                      # Compile firmware to .bin
```

## Step 1 — Wire the blueprint into `main.py`

In your Flask app factory, register the flash blueprint:

```python
# main.py
from firmware.web_flasher.flash_routes import flash_bp

def create_app():
    app = Flask(__name__)
    # ... your existing setup ...
    app.register_blueprint(flash_bp)
    return app
```

Move `firmware/web_flasher/flash.html` to `static/flash.html` so Flask
serves it directly, OR let the blueprint serve it from the firmware/ dir
(it tries both paths).

## Step 2 — Confirm `db.get_user_by_id` exists

The flash routes call `from db import get_user_by_id`. If your `db.py` uses
a different name, edit `_get_user_api_key` in `flash_routes.py`. The
function needs to return a dict with an `api_key` field.

## Step 3 — Ensure these endpoints exist in `api.py`

The bridge firmware calls:

| Method | Path                       | Purpose                          | Auth        |
|--------|----------------------------|----------------------------------|-------------|
| POST   | `/api/sensor/data`         | Receive a sensor reading         | `X-API-Key` |
| GET    | `/api/commands/pending`    | List queued commands             | `X-API-Key` |

Expected POST body for `/api/sensor/data`:
```json
{
  "pod_id": "pod-default",
  "moisture": 47,
  "temperature": 22.3,
  "lux": 850.4,
  "battery": 3.92,
  "boot_count": 12
}
```

Expected GET response for `/api/commands/pending`:
```json
{
  "commands": [
    {"type": "FORCE_READ"},
    {"type": "SET_MODE", "value": "DEMO"}
  ]
}
```

After the bridge forwards a command to the pod, it should call
`POST /api/commands/<id>/ack` (or similar) so you don't re-deliver. You
already have command queueing in your design — just make sure
`/api/commands/pending` is filtered by the API key's user.

## Step 4 — Build the firmware once

```bash
cd firmware/build_scripts
./setup.sh    # one-time
./build.sh    # produces instance/firmware/{pod,bridge,bootloader,partitions}.bin
```

These four .bin files are the SAME for every user. Per-user data
(`api_key`, `server_url`) is written separately at flash time as an NVS
partition image generated by `/api/firmware/nvs-image`.

## Step 5 — Render.com deployment

`instance/firmware/*.bin` should be committed to git (or copied to your
persistent disk during build) so Render serves them. Add to your build
command:

```
pip install -r requirements.txt && cp -r firmware/_built/* instance/firmware/
```

(Or just commit the .bin files — they're ~600 KB each.)

## Step 6 — Add a "Flash my bridge" button to the dashboard

In `static/index.html`, add a link:

```html
<a href="/flash" class="btn">⚡ Flash my bridge</a>
```

The flasher page reads `api_key` from `/api/me/flash-config` (which uses
the existing session) so it auto-fills.

## Pairing flow at first boot

The bridge tries three provisioning methods in order, falling through if each
one fails:

**Tier 1 — NVS-staged credentials (fastest, zero post-flash setup)**
User filled in WiFi on the `/flash` page → SSID and password are baked into
the NVS image alongside the API key → bridge boots, reads NVS, connects.
This is the happy path.

**Tier 2 — BLE provisioning (60s window after boot)**
If NVS has no WiFi creds, the bridge advertises as `EcoAdapt-XXXX` over BLE.
User opens `/pair` on their phone or laptop, picks the device, types SSID +
password, hits Send → bridge writes creds to NVS and reboots into Tier 1.
Web Bluetooth is supported on Chrome/Edge desktop and Chrome Android. It is
**not** supported on iOS Safari — iPhone users fall through to Tier 3.

**Tier 3 — Captive portal AP (last resort)**
If BLE times out, the bridge starts an open AP called `EcoAdapt-Setup`. User
joins it, the captive portal loads automatically (DNS hijack), they enter
WiFi + API key + server URL, hit Save. Bridge reboots into Tier 1.

### Pod pairing (separate, automatic)
Once the bridge is online, the pod just needs to find it:
1. Pod boots, broadcasts a `READING` packet (because `bridge_mac` is FF…).
2. Bridge receives it, learns the pod's MAC, saves it to NVS.
3. Future commands from the cloud get unicast-forwarded to the paired pod.

No user-side pairing UI required for pod ↔ bridge.

## Power consumption notes (pod)

| State                | Current   |
|----------------------|-----------|
| Active read+send     | ~120 mA for ~1.5 s |
| Deep sleep           | ~30 µA   |

In `NORMAL` mode (5min reads / 30s checks) the pod averages roughly 1 mA,
which gives you ~50 days on the 1200mAh LiPo. `ECO` mode roughly doubles
that. `DEMO` mode is for science fair — expect ~1 day battery.

## Things to validate before science fair

1. **NVS image format**: Flash one to a real C3 and verify the firmware
   reads back the correct API key. If it fails, swap in the
   `esp_idf_nvs_partition_gen` tool — same interface, more battle-tested.
2. **HW-390 calibration**: Set `DRY_RAW`/`WET_RAW` in `pod.ino` after
   testing in air vs water.
3. **DS18B20 wiring**: Make sure the 4.7kΩ pullup is between data and 3V3.
4. **VEML7700 voltage**: Confirm it's wired to 3V3 (not 5V) — the XIAO C3
   doesn't have 5V rail anyway.
5. **Web Serial**: Test on Chrome/Edge with a real device before demo day.
